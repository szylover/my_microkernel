\section{本章目标}

\begin{itemize}
  \item 在虚拟地址 \hex{E0000000} 处建立内核堆专用区域
  \item 设计可插拔堆后端接口（\texttt{heap\_ops\_t}）
  \item 实现 first-fit 空闲链表分配器和 slab 分配器

\end{itemize}

\section{背景知识}

PMM 只能分配整页（4KiB），但内核经常需要几十字节的小对象（如 VMA 节点、进程描述符）。
\term{内核堆}在连续虚拟内存上做字节级分配。

\subsection{堆区域规划}

\begin{lstlisting}[style=cstyle]
#define KHEAP_START    0xE0000000u   // 堆虚拟起始
#define KHEAP_MAX_SIZE (256 * 1024 * 1024)  // 256 MiB 上限
#define KHEAP_END      0xF0000000u
\end{lstlisting}

\begin{thinkbox}
直接映射区在 \hex{C0000000}+，bump allocator 在 \hex{D0000000} 附近，
堆在 \hex{E0000000}。为什么要把三个区域分开？
如果它们的地址范围重叠会怎样？
\end{thinkbox}

\section{可插拔后端：\texttt{heap\_ops\_t}}

\begin{lstlisting}[style=cstyle]
typedef struct heap_ops {
    const char* name;
    void  (*init)(void* start, size_t size);
    void* (*alloc)(size_t size);
    void  (*free)(void* ptr);
    void  (*stats)(kheap_stats_t* out);
} heap_ops_t;
\end{lstlisting}

和 \texttt{pmm\_ops\_t} 完全同构。\texttt{kmalloc.c} 是 dispatch 层。

\section{First-Fit 空闲链表}

\subsection{Block Header}

\begin{lstlisting}[style=cstyle]
typedef struct block_header {
    size_t size;                // payload 大小（不含 header）
    int is_free;                // 0=已分配, 1=空闲
    struct block_header* prev;  // 双向链表
    struct block_header* next;
} block_header_t;
\end{lstlisting}

内存布局（单个 block）：\texttt{[header | payload ...]}

\subsection{分配逻辑}

\begin{enumerate}
  \item 从链表头开始遍历，找第一个 \texttt{is\_free \&\& size >= requested} 的块
  \item 如果块远大于请求大小 → \term{拆分}：切成"已分配块 + 新空闲块"
  \item 标记 \texttt{is\_free = 0}，返回 payload 指针
\end{enumerate}

\subsection{释放与合并}

\begin{enumerate}
  \item 标记 \texttt{is\_free = 1}
  \item 检查 \texttt{next}：如果也是 free → \term{向前合并}（合并后 size 增大）
  \item 检查 \texttt{prev}：如果也是 free → \term{向后合并}
\end{enumerate}

\begin{thinkbox}
如果没有合并（coalescing），连续 alloc/free 小块后会出现什么问题？
提示：每个 free block 都很小，但总 free 空间足够——这叫什么现象？
\end{thinkbox}

\section{Slab 分配器}

7 级 cache（32B, 64B, 128B, 256B, 512B, 1024B, 2048B），
每级占整页，bitmap 管理空闲 slot。

优势：零碎片、$O(1)$ 分配、cache 友好。
劣势：大于 2048B 的请求需要回退到其他方案。

\section{kmalloc\_init：初始映射}

\begin{lstlisting}[style=cstyle]
void kmalloc_init(void) {
    for (unsigned i = 0; i < KCONFIG_HEAP_INITIAL_PAGES; i++) {
        uint32_t phys = (uint32_t)pmm_alloc_page();
        vmm_map_page(KHEAP_START + i * 4096, phys, PTE_PRESENT | PTE_WRITABLE);
    }
    g_heap_ops->init((void*)KHEAP_START, KCONFIG_HEAP_INITIAL_PAGES * 4096);
}
\end{lstlisting}

\section{思考与练习}

\begin{exercise}
运行 \texttt{heap alloc 64} 三次，然后 \texttt{heap status} 查看 alloc\_count 和
free\_bytes。计算每次分配的实际开销（payload + header）是多少字节。
\end{exercise}

\begin{exercise}
在 \texttt{kconfig.h} 中切换堆后端为 slab（\texttt{KCONFIG\_HEAP\_BACKEND=1}），
运行 \texttt{heap test}。对比 first-fit 和 slab 的 \texttt{heap status} 输出：
哪个后端的 free\_bytes 更大？为什么？
\end{exercise}

\begin{exercise}
分配一些内存块但故意不释放（模拟内存泄漏），反复分配直到 \texttt{heap alloc}
返回错误。此时 \texttt{heap status} 显示什么？思考：一个健壮的内核如何检测泄漏？
\end{exercise}

\begin{exercise}
（挑战）修改 first-fit 分配器，实现 \term{best-fit} 策略：不取第一个够大的块，
而是取最小的够大的块。对比两种策略的外部碎片表现。
\end{exercise}
